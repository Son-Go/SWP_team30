### ADR-005: Use Layered Architecture (Controller, Service, Repository)

- **Status**: Accepted
- **Context**: The backend needs a clear separation of concerns for maintainability and testability so that the implementation of [QR-001](../../quality-requirements.md#qr-001-patch-gamesid-security-enforcement), [QR-002](../../quality-requirements.md#qr-002-delete-gamesid-security-enforcement), and [QR-003](../../quality-requirements.md#qr-003-public-author-api-contract) can evolve without tightly coupled code.
- **Decision**: Use the layered architecture pattern with Controller, Service, and Repository layers to isolate HTTP handling, business logic, and persistence concerns.
- **Quality requirements addressed**: [QR-001](../../quality-requirements.md#qr-001-patch-gamesid-security-enforcement), [QR-002](../../quality-requirements.md#qr-002-delete-gamesid-security-enforcement), [QR-003](../../quality-requirements.md#qr-003-public-author-api-contract)
- **Consequences**: The codebase becomes easier to understand, test, and extend. The trade-off is that the design introduces some boilerplate and requires discipline in keeping responsibilities separated.
